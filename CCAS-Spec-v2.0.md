## 1. 总则 (General Principles)

### 1.1 设计哲学

本规范旨在构建一套语义直映射的计算体系。核心目标是将人类自然语言逻辑与机器执行逻辑之间的认知摩擦降至最低，同时保持工程上的高效性与安全性。

- 原生中文：从指令集助记符到高级语言语法，全面支持中文字符，不依赖 ASCII 关键字作为语法核心。
- 编码统一：全链路强制使用 UTF-8 编码，消除多编码集转换带来的乱码与存储浪费。
- 生态兼容：通过外部函数接口 (FFI) 兼容现有 C/C++ 二进制库，避免重复造轮子。
- 安全优先：内存安全与并发安全内建于语言核心，而非依赖外部库。

### 1.2 适用范围

本规范适用于以下领域的开发与合规性检查：

1. 编译器开发：符合 CCAS 语法的前端与后端实现。
2. 操作系统适配：提供 CCAS 标准系统调用的内核层。
3. 嵌入式系统：基于 CCAS 指令集映射的固件开发。
4. 上层应用：遵循本规范编写的所有应用程序。

### 1.3 术语定义

- xyc：玄语参考编译器 。
- 语义空格 (Semantic Whitespace)：中文关键字后保留至少一个空格。为解决中文分词歧义制定的词法分隔规则，现调整为警告级别 —— 违反该规则时仅提示警告，不阻断编译 / 执行。
- 原生块 (Unsafe Block)：允许绕过内存安全检查进行底层操作的代码区域。

## 2. 字符与编码层 (Character & Encoding Layer)

### 2.1 源文件编码

- 强制要求：所有源代码文件 (.xy)、头文件 (.xyh)、配置文件必须采用 UTF-8 无 BOM 格式。
- 禁止行为：禁止在源文件中混合使用 GBK、ASCII (作为关键字) 或其他遗留编码。
- 校验机制：编译器 xyc 在词法分析前必须校验文件编码，发现非 UTF-8 文件必须报错终止。

### 2.2 标识符命名

- 字符集：变量名、函数名、类名、模块名允许直接使用 Unicode 中文汉字。
- 命名风格：推荐中文采用自然语言命名法，英文采用驼峰式 (CamelCase) 。

  - 中文：用户年龄，计算订单总额

  - 英文：calcOrderTotal

- 保留字：所有中文关键字（如 若, 则, 函数）为保留字，不可用作标识符。

### 2.3 词法分隔符

中文关键字后保留至少一个空格。为解决中文分词歧义制定的词法分隔规则，现调整为警告级别 —— 违反该规则时仅提示警告，不阻断编译 / 执行：

- 关键字与操作数分隔
- 规则：中文关键字后应保留至少一个空格，以明确关键字与操作数的边界；
- 违规处理：检测到无空格时输出警告信息，标注违规位置，不终止流程；

  - ✅ 合规（推荐）：若 条件 则

  - ⚠️ 警告（不阻断）：若条件则

- 操作数之间分隔
- 规则：中文关键字后应保留至少一个空格，以明确关键字与操作数的边界；
- 违规处理：检测到无空格时输出警告信息，标注违规位置，不终止流程；

  - ✅ 正确：加 甲，乙 或 加 甲 乙

  - ⚠️ 警告（不阻断）：加甲乙

- 标点例外：英文标点 ((), {}, ;) 前后空格遵循通用编程习惯，不强制语义空格，建议保持可读性。

## 3. 指令集映射层 (ISA Mapping Layer)

### 3.1 基础架构

- 物理基座：本规范基于 RISC-V RV64GC 开源指令集物理架构。
- 映射原则：保留 RISC-V 二进制操作码 (Opcode) 不变，仅在汇编语义层进行中文助记符映射。
- 兼容性：生成的机器码必须符合 RISC-V 标准，确保可在标准硬件上运行。

### 3.2 基础指令映射表

| **二进制操作码** **(Hex)** | **英文助记符** | **中文助记符** |  **功能描述**  |
| :------------------------: | :------------: | :------------: | :------------: |
|        `0x33` (ADD)        |     `add`      |      `加`      |   寄存器相加   |
|       `0x03` (LOAD)        |      `lw`      |     `加载`     | 从内存加载数据 |
|       `0x23` (STORE)       |      `sw`      |     `存储`     | 将数据存入内存 |
|      `0x63` (BRANCH)       |     `beq`      |   `若等则跳`   |  条件分支跳转  |
|       `0x6F` (JUMP)        |     `jal`      |     `跳转`     |   无条件跳转   |
|        `0x00` (NOP)        |     `nop`      |    `空操作`    |     无操作     |

### 3.3 寄存器命名规范

物理寄存器编号不变，汇编语言中推荐使用中文别名：

- x0 → 零 (硬连线零)
- x1 → 返址 (返回地址)
- x2 → 栈顶 (栈指针)
- x10–x17 → 参数 1 至 参数 8
- x5–x7 → 暂存 1 至 暂存 3

### 3.4 汇编示例

```asm

# 传统 RISC-V 汇编

add t0, a0, a1

sw t0, 0(s0)

# CCAS 中文汇编

加 暂存 1, 参数 1, 参数 2

存储 暂存 1, 0(栈顶)

```

## 4. 高级语言语法层 (High-Level Syntax Layer)

### 4.1 数据类型

| **英文类型** | **CCAS** **中文类型** | **字节大小** **(64** **位系统\*\***)\*\* |    **说明**    |
| :----------: | :-------------------: | :--------------------------------------: | :------------: |
|    `int`     |        `整数`         |                  4 字节                  |   有符号整型   |
|    `long`    |       `长整数`        |                  8 字节                  |  有符号长整型  |
|   `float`    |       `浮点数`        |                  4 字节                  |   单精度浮点   |
|   `char*`    |        `文本`         |                   可变                   |  UTF-8 字符串  |
|    `bool`    |        `布尔`         |                  1 字节                  |     真/假      |
|    `void`    |       `无返回`        |                  0 字节                  |     无类型     |
|     `T?`     |       `或许<T>`       |                   可变                   | 可能为空的类型 |

### 4.2 控制流结构

#### 条件判断

```

若 (分数 大于 60) 则 {

    打印 ("及格")

} 否则 {

    打印 ("不及格")

}
```

#### 循环结构

```

计数循环

    循环 (i 从 0 到 10) {

    打印 (i)
}


遍历循环

    循环 项 取自 集合 {

    打印 (项)
}

```

#### 函数定义

```

函数 计算面积 (整数 宽，整数 高) 返回 整数 {

    返回 宽 * 高

}

```

### 4.3 内存管理

- 自动管理：默认启用垃圾回收 (GC)，关键字 自动。
- 手动管理：关键性能路径可使用手动管理，关键字 手动。

```

手动 {

    指针 数据 = 分配 (100)

    使用 (数据)

    释放 (数据)

}

```

- 所有权：变量默认拥有所有权，转移需显式声明 (见第 10 章)。

### 4.4 模块与包

- 模块导入：使用 `标准库.内存;`
- 可见性：默认私有，公开需标记 `公开`。

```

公开 函数 外部接口 () {

    ...

}

```

## 5. 系统接口层 (System Interface Layer)

### 5.1 标准库封装

- 禁止直调：应用层禁止直接调用 POSIX 英文 API (如 open, read)。
- 强制封装：必须通过中文标准库封装层调用。

| **功能** | **POSIX 英文** **API** | **CCAS** **中文** **API** |
| :------: | :--------------------: | :-----------------------: |
| 文件打开 |        `open()`        |  `打开文件(路径，模式)`   |
| 文件读取 |        `read()`        |  `读取内容(代号，长度)`   |
| 线程创建 |   `pthread_create()`   |  `开启线程(函数，参数)`   |

### 5.2 错误处理规范

- 禁止错误码：禁止使用整数错误码 (Error Code) 传递错误。
- 结构化异常：必须抛出包含中文描述与建议的异常对象。

```

尝试 {

    打开文件 ("config.txt")

} 捕获 (错误) {

    打印 (错误.描述)  // 例如：“文件未找到”

    打印 (错误.建议)  // 例如：“请检查路径”

}

```

## 6. 工具链规范 (Toolchain Specification)

### 6.1 编译器 (xyc)

- 中间表示：采用 CIR (Chinese Intermediate Representation)，基于 LLVM IR 但元数据保留中文语义。
- 优化策略：针对中文变量名长、重复率高的特点，优化符号表查找效率。
- 错误信息：所有编译错误、警告必须为中文，并指向具体行号与列号。

### 6.2 调试器 (xydb)

- 显示规则：变量监视窗口支持中文、英文变量名正常显示与查看。
- 反汇编：反汇编窗口显示中文助记符 (见 3.2 节)。
- 交互命令：支持中文命令，如 下一步，继续，打印 变量。

### 6.3 包管理器 (仓)

- 名称：仓 (Cang)
- 仓库地址：repo.xy-lang.org
- 依赖文件：依赖清单.txt

```json
{
  "项目名称": "电商系统",

  "依赖": {
    "网络库": "1.0.0",

    "数据库驱动": "2.3.1"
  }
}
```

## 7. 生态兼容与互操作 (Interop)

### 7.1 外部函数接口 (FFI)

- 目的：解决生态隔离，允许调用 C/C++ 动态库。

```

外部 库 "libcrypto.so" {

    函数 加密 (文本 数据，文本 密钥) 返回 文本

}

```

- 命名映射：调用外部库时，允许保留英文函数名，但建议在中文层包裹一层别名。

### 7.2 数据交换格式

- 默认格式：JSON 或 XML。
- CJSON 标准：支持中文键名无需转义。

```json
{
  "用户 ID": 1001,

  "登录状态": true
}
```

## 8. 附录：实现路线图

1. 阶段一 (原型)：基于 LLVM 开发前端解析器，实现中文到 LLVM IR 的转换，后端复用现有编译器。
2. 阶段二 (工具)：开发 VS Code 插件，提供中文语法高亮、自动补全、输入法集成。
3. 阶段三 (系统)：基于 Linux 内核开发中文系统调用层，实现 xyc 原生编译的操作系统。
4. 阶段四 (生态)：建立中文开源社区，鼓励用中文编写基础库 (网络、UI、数据库)。

## 9. 自举与过渡策略 (Bootstrapping & Transition)

### 9.1 引导编译器

- 现状承认：第一代中文编译器 (xyc-v1) 必须使用成熟的英文工具链（如 C++/LLVM）构建。
- 自举目标：xyc-v2 必须使用 xyc-v1 编译生成，实现完全自举。
- 过渡期规范：

  - 在自举完成前，允许核心底层代码（内核、编译器前端）使用中英混合模式。

  - 混合模式标记：使用 @extern 标记英文代码块，编译器需忽略其语义检查，仅作二进制链接。

  - 截止日期：设定“纯中文日”，此后所有提交必须通过中文语法检查。

### 9.2 渐进式迁移

- 允许现有 C/C++ 库通过 FFI 包装器接入。
- 包装器必须提供完整的中文文档和中文类型映射，禁止直接暴露英文符号给应用层。

## 10. 内存与安全模型 (Memory & Safety Model)

### 10.1 所有权语义

借鉴现代内存安全理念，引入中文所有权概念：

- 所有权：变量独占资源。
- 借用：临时引用，不可修改。
- 可变借用：临时引用，可修改。
- 生命周期：显式标注引用有效范围。

示例：

```

函数 处理数据 (借用 文本 输入) 返回 整数 {

    // 输入只读，编译期确保不被修改

    返回 输入.长度

}

```

### 10.2 内存操作限制

- 禁止直接使用指针算术运算（如 ptr + 1），除非标记为 不安全 块。

```

不安全 {

    // 仅限底层驱动开发，需人工审计

    原始指针 地址 = 获取硬件地址 ()

}

```

### 10.3 权限沙箱

- 程序启动时默认无权限。
- 必须在 manifest 中声明所需权限：

```

权限 {

    允许 读取 文件 "~/data"

    允许 访问 网络 "api.example.com"

    禁止 访问 摄像头

}

```

## 11. 并发与异步模型 (Concurrency & Async)

### 11.1 线程与协程

- 线程：操作系统级并行。
- 协程：用户态轻量级并发。

```

启动 线程 { 执行任务 () }

启动 协程 { 异步任务 () }

```

### 11.2 异步等待

- 关键字：异步 (async), 等待 (await)。
- 规范：等待 关键字只能在标记为 异步 的函数中使用。

```

异步 函数 获取用户数据 () {

    数据 = 等待 网络.请求 ("GET /user")

    返回 数据

}

```

### 11.3 同步原语

- 锁 (Mutex), 读写锁 (RwLock), 通道 (Channel)。
- 推荐使用 通道 进行线程间通信，避免共享内存竞争。

```

通道 消息通道 = 创建通道 ()

发送 消息通道，"你好"

内容 = 接收 消息通道

```

## 12. 测试与验证规范 (Testing & Verification)

### 12.1 内置测试框架

- 语言原生支持测试块，无需第三方库。

```

测试 函数 测试加法 () {

    验证 等于 (加 (1, 2), 3)

}

```

运行命令：中文测试 运行

### 12.2 文档即测试 (Doc Tests)

- 代码注释中的示例代码必须可编译、可运行。
- 编译器在构建过程中会自动提取注释中的代码片段进行测试。

### 12.3 形式化验证接口

- 对于关键安全模块，支持接入形式化验证工具。
- 关键字：契约 (Contract)。

```

函数 除法 (整数 被除数，整数 除数) 返回 整数 契约 {

    前置 条件：除数 != 0

    后置 条件：返回 值 * 除数 == 被除数

}

```

## 13. 治理与演进 (Governance & Evolution)

### 13.1 中文计算委员会 (CCC)

- 设立常设机构负责规范维护。
- 职责：审核新关键字、解决语义歧义、维护标准库。
- 组成：由编译器作者、语言设计师、安全专家组成。

### 13.2 RFC 流程 (Request for Comments)

- 任何语法变更必须经过 RFC 流程。
- 阶段：草案 -> 公众评议 -> 试点实现 -> 正式合并。
- 语义稳定性：一旦关键字进入稳定版，禁止随意更改含义，确保向后兼容。

### 13.3 歧义仲裁原则

当中文词汇存在多义性时，遵循以下优先级：

1. 计算机科学通用译名（如“队列”优于“排队”）。
2. 自然语言常用义（如“打印”优于“输出”）。
3. 上下文推断（由编译器根据类型推断）。

## 14. 编码风格指南 (Style Guide)

### 14.1 借词白名单 (实用主义)

- 逻辑动词：必须使用中文 (如 若, 返回, 打印, 循环)。
- 技术名词/缩写：保留英文 (如 HTTP, JSON, CPU, ptr, ID)。

  - ✅ 有效：用户 ID, HTTP 请求, ptr

  - ❌ 无效：用户标识符, 超文本传输协议请求 (过于冗长)

### 14.2 标点使用规范

- 代码结构：必须使用英文标点 (;, (), {}, :) 以减少输入法切换和对齐问题。
- 字符串/注释：建议使用中文标点 (，。“”) 以符合自然语言习惯。
- 空格：中文关键字后加空格，英文标点周围遵循常规编程风格。

### 14.3 代码示例规范

- 所有官方文档示例必须遵循 CCAS 语法。
- 混合英文关键字仅允许在 外部 块 (FFI) 或 不安全 块中出现。
- 示例必须包含中文注释，解释关键逻辑。
