# 玄语编译器项目编码规范

## 一、规范目的

为保证玄语编译器代码的可读性、可维护性、一致性，降低团队协作成本，同时适配中文关键字/标识符的特色，特制定本编码规范。本规范适用于玄语编译器（XY编译器）的所有核心代码（词法分析、语法分析、代码生成、模块系统等）及配套工具链。

## 二、基础编码约定

### 2.1 字符编码与文件格式

1. **字符编码**：所有源码文件统一使用 `UTF-8` 编码（无BOM），确保中文关键字、标识符、注释的正常显示。
2. **换行符**：统一使用 `LF`（\n）作为换行符，禁止使用 `CRLF`（\r\n）。
3. **文件结尾**：每个源码文件末尾必须保留一个空行，避免编译工具报“文件未结束”类警告。
4. **文件命名**：
   - 源文件命名使用「小写英文+下划线」风格（如 `lexer_analysis.c`、`module_loader.xy`）；
   - 中文标识符文件可兼容「小写中文+下划线」（如 `语法分析器.xy`），但优先推荐英文命名以适配跨平台工具链；
   - 测试文件在原文件名后加 `_test`（如 `code_gen_test.xy`）。

### 2.2 缩进与空格

1. **缩进**：统一使用 4 个空格缩进，禁止使用 Tab 字符（需配置编辑器将 Tab 自动转为 4 空格）。
2. **空格使用原则**：
   - 二元运算符（`+`、`=`、`&&` 等）两侧各加 1 个空格（如 `a = b + c`，而非 `a=b+c`）；
   - 逗号、分号后加 1 个空格（如 `func(a, b);`，而非 `func(a,b);`）；
   - 括号内侧无空格（如 `if (x > 0)`，而非 `if ( x > 0 )`）；
   - 函数名与左括号之间无空格（如 `compile()`，而非 `compile ()`）。

### 2.3 注释规范

1. **注释语言**：优先使用中文注释（适配项目中文特性），关键逻辑必须加注释，禁止无意义注释（如 `// 加1` 注释 `i++`）。
2. **注释类型**：
   - **文件头注释**：每个源码文件开头必须包含版权、作者、功能描述、修改记录（示例如下）；
   ```c
   /*
    * 玄语编译器 - 词法分析模块
    * 功能：实现中文关键字识别、标识符解析、词法错误检测
    * 作者：XXX
    * 版本：v0.2
    * 修改记录：
    * 2026-03-21 - 完成中文字符串词法解析逻辑
    */
   ```

   - **函数/方法注释**：核心函数（如语法分析、代码生成）必须说明入参、出参、功能、异常场景；
   - **行内注释**：仅用于解释复杂逻辑（如算法关键点、特殊兼容处理），避免每行都加注释。
3. **注释更新**：修改代码时必须同步更新关联注释，禁止注释与代码逻辑不一致。

## 三、语言特性约定（适配玄语中文特性）

### 3.1 标识符与关键字

1. **中文标识符规则**：
   - 中文标识符需表意清晰（如 `导入模块` 而非 `做导入`），避免生僻字或歧义词汇；
   - 混合中英文标识符时，英文部分小写，中文部分与英文之间用下划线分隔（如 `导入_module`）；
   - 禁止使用纯数字、特殊符号（除下划线）作为标识符。
2. **关键字使用**：
   - 严格遵循玄语定义的中文关键字（如「引入」「函数」「返回」），禁止自定义与关键字冲突的标识符；
   - 关键字书写统一使用简体中文，禁止繁体（如用「函数」而非「函數」）。
3. **命名语义化**：
   - 变量名：采用「名词/形容词+名词」（如 `语法树节点`、`llvm_ir_buffer`）；
   - 函数名：采用「动词+名词」（如 `生成IR代码`、`解析导入语句`）；
   - 常量名：全大写，下划线分隔（如 `MAX_TOKEN_LENGTH`、`默认模块路径`）。

### 3.2 代码结构约定

1. **函数/方法**：
   - 单个函数代码行数不超过 80 行，超过则拆分（如将复杂的代码生成逻辑拆分为多个子函数）；
   - 函数入参不超过 5 个，超过则封装为结构体/对象（如 `编译上下文` 结构体）；
   - 禁止嵌套超过 4 层（如 if-else、循环嵌套），超过则提取为子函数。
2. **模块导入**：
   - 导入语句统一放在文件顶部，按「标准库→自研模块→第三方模块」排序；
   - 禁止导入未使用的模块（避免编译冗余）；
   - 中文导入语句格式：`引入 "模块路径" 作为 别名;`（如 `引入 "语法分析/节点.xy" 作为 语法节点;`）。
3. **错误处理**：
   - 所有可能出错的环节（如文件读取、语法解析）必须有错误捕获和提示；
   - 错误提示需包含「错误类型+位置+原因」（如 `语法错误：第10行第5列 - 未闭合的中文字符串`）；
   - 禁止使用 `exit()` 直接终止程序，需返回错误码并向上层传递。

## 四、工程化约定

### 4.1 版本控制（Git）

1. **提交信息**：
   - 提交信息格式：`[模块名] 动作：描述`（如 `[词法分析] 修复：中文关键字"引入"识别错误`）；
   - 提交信息使用中文，简洁明了（不超过 50 字），禁止无意义提交（如「改bug」「更新代码」）。
2. **分支管理**：
   - `main` 分支：仅存放稳定可发布版本，禁止直接提交；
   - `dev` 分支：开发主分支，所有功能分支需合并到 dev 后再提测；
   - 功能分支：命名格式 `feature/模块-功能`（如 `feature/代码生成-llvm_ir优化`）；
   - 修复分支：命名格式 `fix/模块-问题`（如 `fix/模块导入-路径解析错误`）。

### 4.2 测试规范

1. **自展测试**：每次核心逻辑修改后，必须执行自展测试（验证编译器能否编译自身）；
2. **单元测试**：
   - 词法分析、语法分析、代码生成等核心模块需编写单元测试，覆盖率不低于 80%；
   - 测试用例需覆盖「正常场景+异常场景」（如合法中文标识符、非法字符、嵌套导入）；
3. **测试命名**：测试函数命名格式 `test_模块_功能_场景`（如 `test_语法分析_中文关键字_边界值`）。

### 4.3 兼容性约定

1. **跨平台**：代码需兼容 Linux/macOS，核心逻辑禁止依赖特定系统的 API（如 Windows 的 `Win32` 接口）；
2. **LLVM 版本**：统一使用指定版本的 LLVM（如 16.0），代码中避免使用 LLVM 私有 API，降低版本升级成本；
3. **中文环境**：兼容不同操作系统的中文显示（如 Linux 下的 LC_ALL 配置、macOS 的中文编码）。

## 五、规范落地与执行

1. **工具校验**：使用格式化工具（如 clang-format、prettier）统一代码格式，提交前执行 lint 检查；
2. **代码评审**：所有合并到 dev 分支的代码必须经过至少 1 名核心开发者评审，重点检查规范符合性；
3. **规范更新**：本规范随项目迭代更新，更新后需同步通知所有开发者，确保认知一致。

### 总结

1. 玄语编译器编码规范核心围绕「中文特性适配+工程化一致性」，优先保证代码可读性和自展编译的稳定性；
2. 基础约定（编码、缩进、注释）是团队协作的基础，语言特性约定（中文标识符、模块导入）适配玄语核心特色；
3. 工程化约定（版本控制、测试）是保障编译器迭代质量的关键，需严格执行。

可根据项目后续迭代（如新增运行时、优化器模块）补充对应模块的专项规范，确保规范始终适配项目发展。
